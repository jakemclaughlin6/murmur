## Decisions Made

See frontmatter `key-decisions`. Two worth expanding:

**D-02-03-A: FK pragma in beforeOpen is mandatory, not optional.** The first run of the migration test failed on the cascade-delete assertion: inserting a Book + Chapter, deleting the Book, and expecting the Chapter to be gone produced `[Chapter:Chapter(id: 1, …)]` instead. The cause is that SQLite ships with `foreign_keys` disabled by default, and the pragma is a per-connection setting, so it must be issued in Drift's `beforeOpen` hook. This is a well-known Drift gotcha but was not in the plan. Without this pragma, D-03's `ON DELETE CASCADE` claim is a lie — Plan 08's persistence-across-restart test and any future "delete book" user flow would produce orphan chapter rows and silently grow the DB. Added in `beforeOpen` rather than deferring: tracked as Rule 2 deviation below.

**D-02-03-B: Hand-authored migration test, generated snapshots.** The plan anticipated that `drift_dev schema generate --data-classes --companions` would emit both schema snapshots AND a round-trip migration test file. It emitted only the snapshots — no test file. The reason (inferred from drift's source): when v1 has zero entities, there is nothing for the generator's default `verifySelf` template to diff against. Rather than temporarily reverting v1 to add a fake entity just to get the generator to emit a test (which would pollute the v1 schema baseline), I hand-authored the test using the same public API the generator would have used (`SchemaVerifier` + `GeneratedHelper`). The generated snapshots (`DatabaseAtV1`, `DatabaseAtV2`) remain the source of truth for the shapes being validated; only the test body is hand-written.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing critical functionality] `PRAGMA foreign_keys = ON` not set**
- **Found during:** Task 2 (first run of the migration test's cascade-delete assertion)
- **Issue:** The plan defined `chapters.bookId` with `onDelete: KeyAction.cascade`, but SQLite's `ON DELETE CASCADE` only fires when `PRAGMA foreign_keys = ON` is active on the connection. SQLite defaults to `OFF`. Drift does not set this pragma automatically. The cascade-delete assertion in the migration test failed with `Expected: empty / Actual: [Chapter(id: 1, …)]`, proving the declared FK was silently ignored at runtime. This is not a test-only issue — in production, deleting a book from the library would leave orphan chapter rows that grow the DB indefinitely and break Plan 08's persistence contract.
- **Fix:** Added a `beforeOpen` hook in `AppDatabase.migration` that runs `customStatement('PRAGMA foreign_keys = ON')`. The pragma is a per-connection setting so `beforeOpen` is the Drift-idiomatic place for it — it re-issues on every new connection (including in-memory test databases).
- **Files modified:** `lib/core/db/app_database.dart`
- **Verification:** Migration test's cascade-delete assertion now passes (0 chapters remain after book delete). `flutter test` full suite 55 pass / 14 skipped / 0 fail. Plan 08's persistence-across-restart test inherits this guard automatically.
- **Committed in:** `ee2d686` (Task 2)

**2. [Rule 3 - Blocking] `drift_dev schema generate` did not emit a migration test file**
- **Found during:** Task 2 after `dart run drift_dev schema generate drift_schemas/ test/generated_migrations/ --data-classes --companions`
- **Issue:** The plan said this command would write `test/generated_migrations/schema_v1_to_v2_test.dart` with a `verifySelf()` round-trip. It wrote only `schema.dart`, `schema_v1.dart`, `schema_v2.dart` — no test file. The most plausible reason is that v1 has zero entities (confirmed in `drift_schemas/drift_schema_v1.json`: `"entities": []`), so there is nothing for the default `verifySelf` template to diff against and the generator silently omits the test.
- **Fix:** Hand-authored `test/generated_migrations/schema_v1_to_v2_test.dart` using the same public API the generator would have used: `SchemaVerifier(GeneratedHelper())`, `startAt(1)`, `migrateAndValidate(db, 2)`. The hand-authored test also adds a second case that inserts a Book + Chapter and asserts cascade delete — which is what caught the FK pragma issue above. The generated `schema.dart`/`schema_v1.dart`/`schema_v2.dart` snapshots remain the source of truth for what the two schemas look like; only the test body is hand-written. Per D-04, no hand-written migration SQL exists anywhere — only test glue.
- **Files modified:** `test/generated_migrations/schema_v1_to_v2_test.dart` (new, 99 lines)
- **Verification:** Both tests pass. `SchemaVerifier.migrateAndValidate(db, 2)` is the exact API drift-generated `verifySelf` tests call under the hood, so this is functionally equivalent — just less generated.
- **Committed in:** `ee2d686` (Task 2)

**3. [Rule 1 - Bug] Pre-existing app_database_test asserted schemaVersion=1 and zero tables**
- **Found during:** Task 1 (between writing the new tables and running the suite; pre-emptively fixed per advisor guidance)
- **Issue:** `test/db/app_database_test.dart` from Phase 1 has three assertions: (a) `schemaVersion is 1`, (b) `has zero user tables (D-15)`, (c) `close()` succeeds on an empty DB. After bumping to v2 and creating both tables via `onCreate`, (a) and (b) both break. The plan's acceptance criteria says "flutter test still passes" — this test was the only obstacle.
- **Fix:** Updated the three test assertions: schemaVersion 1 → 2, zero-tables → `containsAll(['books', 'chapters'])`, group description updated from FND-04 to D-04/D-05/D-03. The "close() succeeds" test still passes unchanged.
- **Files modified:** `test/db/app_database_test.dart`
- **Verification:** `flutter test` all green.
- **Committed in:** `90b28a8` (Task 1)
