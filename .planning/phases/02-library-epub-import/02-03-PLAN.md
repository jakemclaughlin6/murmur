<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Existing user DB → upgraded DB | On app update, user has a v1 DB with zero tables; migration writes new schema. Data loss is the primary risk. |
| Drift query layer → SQLite | Drift generates parameterized queries; untrusted input from Plans 05-07 crosses this boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Tampering | books.file_path column | mitigate | `.unique()` constraint prevents duplicate imports of the same file. Downstream Plan 05 will ALSO validate file paths are inside the app documents dir (defense in depth against path traversal). |
| T-02-03-02 | Denial of Service | chapters.blocks_json unbounded TEXT | accept | SQLite TEXT columns are bounded by SQLITE_MAX_LENGTH (~1GB default). The parser (Plan 04) enforces a practical per-chapter size limit. This plan's schema cannot enforce byte limits without `CHECK` constraints which drift_dev does not ergonomically expose. |
| T-02-03-03 | Tampering | onUpgrade migration | mitigate | Generated `stepByStep` test (`verifySelf`) round-trips the migration, preventing silent schema drift. v2 schema JSON is committed so any unintended change to Phase 2 tables fails the test. |
| T-02-03-04 | Information Disclosure | books.file_path stored as plaintext | accept | The path IS the file the user owns; no PII beyond what they already have on disk. App sandbox protects the DB file from other apps. |
</threat_model>
