<objective>
Build the pure-Dart EPUB parser that transforms EPUB bytes into a `ParseResult` with metadata and a `List<Chapter>` where each chapter holds a `List<Block>`. Run it in an Isolate. Reject DRM-protected EPUBs up front. Handle malformed XHTML per-chapter (graceful degradation, not whole-book failure).

Purpose: LIB-03 requires parsing a 15-EPUB corpus. LIB-04 requires rejecting corrupt/DRM books. D-01/D-02/D-13 lock the full Block IR at import time in an Isolate. This plan delivers the parser core — the riskiest work in Phase 2 because it depends on `epubx` (unverified assumption A2 from RESEARCH.md, validated in Plan 01) and `package:html` leniency against real-world XHTML. The parser is pure Dart (no Flutter imports) so it can cross the isolate boundary and be unit-tested without Flutter runtime.

```dart
/// Result of parsing an EPUB file.
/// Constructed by the parser, consumed by the import service (Plan 05).
class ParseResult {
  final String title;
  final String? author;
  /// Raw cover image bytes (JPEG/PNG/etc), null if the EPUB has no cover.
  /// Plan 05 writes these to ${appDocumentsDir}/covers/{bookId}.jpg per D-06.
  final List<int>? coverBytes;
  /// MIME type of coverBytes (e.g., "image/jpeg"), null when coverBytes is null.
  final String? coverMimeType;
  final List<ParsedChapter> chapters;
  /// Per-chapter parse warnings (malformed XHTML, etc).
  /// Empty list = clean parse.
  final List<ChapterError> errors;

  const ParseResult({
    required this.title,
    required this.author,
    required this.coverBytes,
    required this.coverMimeType,
    required this.chapters,
    required this.errors,
  });
}

class ParsedChapter {
  final int orderIndex;
  final String? title;
  final List<Block> blocks;
  const ParsedChapter({
    required this.orderIndex,
    required this.title,
    required this.blocks,
  });
}

class ChapterError {
  final int orderIndex;
  final String message;
  const ChapterError(this.orderIndex, this.message);
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User's file bytes → parser | Untrusted EPUB bytes from file system, Share intent, or document picker enter the parser. |
| Parser → Block IR | Parser output becomes application-trusted data structures stored in the DB. |
| Main isolate ↔ parse isolate | Bytes cross the isolate boundary; exceptions must be marshaled. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04-01 | Denial of Service | ZipDecoder on malicious EPUB (zip bomb) | mitigate | `package:archive` has bounded decompression; additionally, the parser runs in an isolate so even a stall cannot freeze the UI thread. Plan 05 (import service) applies a 30s timeout on `parseEpubInIsolate`. |
| T-02-04-02 | Tampering | DRM bypass via malformed META-INF | mitigate | `detectDrm` checks BOTH `encryption.xml` AND `rights.xml` via exact-path match in the archive file list. Case-insensitive compare not needed (EPUB spec mandates exact paths). Per LIB-04. |
| T-02-04-03 | Information Disclosure | XHTML injection via malformed tags | mitigate | `package:html` is a tolerant HTML5 parser; it cannot be tricked into executing code (no JS, no DOM eval — we only read text). Output is plain strings stored in Block.text. |
| T-02-04-04 | Information Disclosure | Image cover path traversal (`../../etc/passwd`) | mitigate | Cover bytes are extracted FROM the archive by EPUB-internal href and the raw BYTES are returned to Plan 05. Plan 05 controls the output path (`${appDocumentsDir}/covers/{bookId}.jpg`). The EPUB cannot influence the destination file name. |
| T-02-04-05 | Denial of Service | Oversized cover image (decompression bomb) | accept | Plan 05 will impose a size cap on coverBytes before writing to disk. This plan emits raw bytes; the cap is a downstream concern. Cap value: 10 MB (enforced in Plan 05). |
| T-02-04-06 | Tampering | Exception type loss across isolate boundary | mitigate | Test 2 and Test 3 in Task 3 explicitly verify DrmDetectedException and EpubParseException marshal correctly. If they don't, fallback to message-string assertions documented in tests. |
</threat_model>
