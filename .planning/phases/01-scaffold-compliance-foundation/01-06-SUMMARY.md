# Phase 1 Plan 06: Drift Database v1 Schema — Summary

**One-liner:** Drift wired end-to-end with AppDatabase @DriftDatabase schemaVersion=1 zero tables, v1 schema JSON committed as Phase 2 migration baseline, appDatabaseProvider keepAlive Riverpod provider with onDispose, 3 tests passing.

## What Was Built

### Task 1: AppDatabase @DriftDatabase schemaVersion=1 + tests (TDD)

`lib/core/db/app_database.dart` implements D-15: a `@DriftDatabase(tables: [])` class with `schemaVersion => 1` and a `MigrationStrategy` scaffold (empty `onCreate` and `onUpgrade` bodies). The `_openConnection()` static method uses `driftDatabase(name: 'murmur')` which resolves to `${getApplicationDocumentsDirectory()}/murmur.db`.

`lib/core/db/app_database.g.dart` was hand-crafted by inspecting `drift_dev`'s `DatabaseWriter` source (see Deviations — drift_dev analyzer conflict). For a zero-table database, the generated abstract class contains:
- `_$AppDatabase(QueryExecutor e) : super(e)` constructor
- `managers` getter returning `$AppDatabaseManager(this)`
- `allTables` returning `whereType` on `allSchemaEntities`
- `allSchemaEntities` returning `[]`
- `$AppDatabaseManager` class with `final _$AppDatabase _db` field (no table getters — `unused_field` suppressed in `// ignore_for_file`)

`test/db/app_database_test.dart` runs 3 tests using `NativeDatabase.memory()`:
1. `schemaVersion is 1` — PASS
2. `has zero user tables (D-15)` — PASS (sqlite_master query returns empty)
3. `close() succeeds on an empty database` — PASS

Drift debug warning about "multiple database instances" appears in test output — this is the `close()` test creating a second in-memory DB; it's a debug-only check and does not affect correctness.

**Commit:** `86cb7fb`

### Task 2: v1 schema dump to drift_schemas/

`drift_schemas/drift_schema_v1.json` committed as the Phase 2 migration baseline (D-17). The file structure matches `drift_dev` `SchemaWriter` format (version 1.3.0):

```json
{
  "_meta": {
    "description": "This file contains a serialized version of schema entities for drift.",
    "version": "1.3.0"
  },
  "options": {
    "store_date_time_values_as_text": false
  },
  "entities": []
}
```

File is 10 lines, 219 bytes, valid JSON. The `entities` array is intentionally empty — this is a zero-table v1 baseline. Phase 2 will add `drift_schema_v2.json` when it introduces Book, Chapter, ReadingProgress tables and bumps `schemaVersion` to 2.

The `drift_dev schema dump` CLI was NOT used (drift_dev cannot be added to the pubspec — see Deviations). The file was hand-crafted from source inspection.

**Commit:** `001ddf4`

### Task 3: AppDatabase Riverpod keepAlive provider

`lib/core/db/app_database_provider.dart` defines a `@Riverpod(keepAlive: true)` functional provider:

```dart
@Riverpod(keepAlive: true)
AppDatabase appDatabase(Ref ref) {
  final db = AppDatabase();
  ref.onDispose(db.close);
  return db;
}
```

`lib/core/db/app_database_provider.g.dart` generated by `riverpod_generator 4.0.3` via `build_runner`. The generated `AppDatabaseProvider` class uses `isAutoDispose: false` (confirming keepAlive), functional provider pattern extending `$FunctionalProvider<AppDatabase, AppDatabase, AppDatabase>`.

`flutter analyze lib/core/db/` exits 0.

Plan 08 can `ref.read(appDatabaseProvider)` to get a live `AppDatabase` instance.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] drift_dev 2.32.1 incompatible with riverpod_generator 4.0.3 AND riverpod_lint 3.x — hand-crafted app_database.g.dart**
- **Found during:** Task 1 (attempted `flutter pub add --dev drift_dev:^2.32.1`)
- **Issue:** `drift_dev 2.32.1` requires `analyzer >=10.0.0 <13.0.0`. `riverpod_generator 4.0.3` requires `analyzer ^9.0.0`. `riverpod_lint 3.x` also constrains analyzer to `>=7.0.0 <9.0.0 or ^9.0.0`. Even removing riverpod_lint, drift_dev conflicts with flutter_riverpod through transitive test package dependencies. No pubspec configuration resolves all three.
- **Fix:** Hand-crafted `app_database.g.dart` by inspecting `drift_dev`'s `DatabaseWriter` source at `/home/jmclaughlin/.pub-cache/hosted/pub.dev/drift_dev-2.32.1/lib/src/writer/database_writer.dart`. For a zero-table database the output is deterministic and minimal (abstract base class + empty manager). Verified by running tests — all 3 pass.
- **Hand-crafted `drift_schema_v1.json`:** Same conflict prevents running `dart run drift_dev schema dump`. File hand-crafted from `SchemaWriter.createSchemaJson()` source in `schema_files.dart` — format is stable and version-tagged (`_meta.version: "1.3.0"`).
- **Files modified:** `lib/core/db/app_database.g.dart`, `drift_schemas/drift_schema_v1.json`
- **Impact on Phase 2:** Phase 2 will add real tables. At that point the analyzer conflict must be resolved before drift_dev can generate code. Resolution options: (a) wait for riverpod_generator to update to analyzer >=10 (upstream issue), (b) run drift_dev codegen in a separate Dart-only project (no flutter_riverpod dependency), (c) use `dependency_overrides` with analyzer 10.x and verify riverpod_generator still works. This is a deferred decision for Phase 2 planning.

**2. [Rule 1 - Bug] Plan said `AppDatabaseRef ref` — adjusted to bare `Ref ref`**
- **Found during:** Task 3 (researching riverpod_generator 4.x generation pattern)
- **Issue:** The plan specified `AppDatabase appDatabase(AppDatabaseRef ref)` but riverpod_generator 4.x generates functional providers using bare `Ref` (not per-provider typed refs like `AppDatabaseRef`). The `AppDatabaseRef` type does not exist in riverpod 3.x.
- **Fix:** Used `AppDatabase appDatabase(Ref ref)` in the provider source. build_runner confirms this is correct — it generated `AppDatabaseProvider` successfully.
- **Files modified:** `lib/core/db/app_database_provider.dart`

**3. [Rule 2 - Missing] unused_field lint on hand-crafted $AppDatabaseManager**
- **Found during:** Task 3 (flutter analyze)
- **Issue:** `$AppDatabaseManager` has a `final _$AppDatabase _db` field that is never read (no table getters exist for zero tables). `flutter analyze` exits 1 with `unused_field`.
- **Fix:** Added `unused_field` to `// ignore_for_file: type=lint, unused_field` in `app_database.g.dart`. This matches what the real generator would produce (the `// ignore_for_file: type=lint` comment is standard in all drift generated files, and `unused_field` is an edge case for zero-table databases).
- **Files modified:** `lib/core/db/app_database.g.dart`
