  <behavior>
    - `MurmurAudioHandler`:
      - `play()` → `ref.read(playbackStateProvider.notifier).setPlaying(true)`; updates `playbackState.add(...)` with `playing: true`, controls `[MediaControl.pause, MediaControl.skipToNext]`, processingState `ready`.
      - `pause()` → `setPlaying(false)`; updates playbackState with `playing: false`, controls `[MediaControl.play, MediaControl.skipToNext]`.
      - `skipToNext()` → advance chapterIdx by +1 via `playbackStateProvider.notifier.setChapter(current + 1)`; if at last chapter, no-op.
      - `updateMediaItem({bookId, bookTitle, author, chapterName, coverPath})` → `mediaItem.add(MediaItem(id: bookId, title: bookTitle, artist: author, album: chapterName, artUri: coverPath != null ? Uri.file(coverPath) : null))`. If coverPath is null OR file doesn't exist, use no artUri (audio_service falls back to app icon per platform norms).
      - `skipToPrevious()`, `seek()`: intentionally unimplemented (SeekHandler mixin still included but with no-op seeks — verify via test).
    - `configureAudioSession()`:
      - `final session = await AudioSession.instance; await session.configure(AudioSessionConfiguration.speech());`
      - Subscribe to `interruptionEventStream`:
        - `event.begin == true` → call `handler.pause()`
        - `event.begin == false` and `event.type == AudioInterruptionType.pause || AudioInterruptionType.unknown` → call `handler.play()` (auto-resume per D-19)
        - `event.type == AudioInterruptionType.duck` under speech category is unusual — treat as pause-resume for consistency (no ducking, D-19).
      - Subscribe to `becomingNoisyEventStream`: always `handler.pause()` (headphone unplug / BT disconnect).
    - Provider `audioServiceProvider`:
      - Async singleton that calls `AudioService.init(builder: () => MurmurAudioHandler(ref: ref), config: AudioServiceConfig(...))` once.
      - Config: `androidNotificationChannelId: 'dev.jmclaughlin.murmur.audio'`, `androidNotificationChannelName: 'Playback'`, `androidNotificationOngoing: true`, `androidStopForegroundOnPause: true`, `notificationColor` (reader theme accent), `androidNotificationIcon: 'mipmap/ic_launcher'` (Phase 1 asset).
      - After init, calls `configureAudioSession(handler)`.
      - Subscribes to `playbackStateProvider` — on bookId/chapterIdx change, calls `handler.updateMediaItem(...)` with metadata pulled from the books Drift table.
    - The audio handler DOES NOT drive synthesis — TtsQueue (04-06) remains the synth engine. The handler is purely a lock-screen/background control bridge. When the user taps play on the lock screen, that flows into `playbackStateProvider.setPlaying(true)`, which the queue driver reacts to.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08-01 | Spoofing | Media-button intent replay by other apps | accept | Android's MediaSession framework is trusted; audio_service 0.18.18 delegates to platform MediaSession which handles replay defense. |
| T-04-08-02 | Denial of Service | Audio session hijack by another media app | mitigate | Session configured as `speech()` — respects AudioFocus; pauses cleanly on loss. |
| T-04-08-03 | Information Disclosure | Lock-screen shows book title to shoulder-surfers | accept | User control: they can pause or use airplane-mode-equivalent (Android's "hide sensitive content" lock screen setting) to suppress. Not an app-level threat. |
| T-04-08-04 | Tampering | Android 14 killing the service mid-playback | mitigate | `foregroundServiceType="mediaPlayback"` + `FOREGROUND_SERVICE_MEDIA_PLAYBACK` permission present (Pitfall 6); compliance test asserts both. |
| T-04-08-05 | Information Disclosure | Cover art leaking private book title via notification | accept | User imported the book; notification is user-visible state reflecting their active session. |
</threat_model>
